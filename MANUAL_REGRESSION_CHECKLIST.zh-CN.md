# ShardKey 手工回归清单

适用范围：当前“离线问答解锁器”版本，重点覆盖以下改动：

- Solve 页状态机修复
- 文案与安全边界澄清
- 每题独立校验标签 `xorTags`
- 新 challenge 优先走“先验题再恢复”路径
- 新 challenge 的标签校验与线性恢复

## 测试环境

至少覆盖两种打开方式：

1. `file://` 直接打开 `index.html`
2. `python3 -m http.server 8000` 后访问 `http://localhost:8000`

建议浏览器：最新版 Chrome / Edge / Firefox 中任意一种。

## 一、创建挑战

### 1. 基础创建

1. 打开创建页。
2. 输入一个秘密，例如：`hello-secret-123`。
3. 添加 3 个问题，每题 1 个正确答案。
4. 设置门限为 `2`。
5. 点击“生成挑战”。

预期结果：

- 成功生成挑战。
- 结果区出现分享链接或 JSON 下载按钮。
- 不报错、不出现空白结果。

### 2. 多备选答案创建

1. 创建一个 3 题门限 `2` 的挑战。
2. 其中至少 1 个问题添加 2~3 个备选答案。
3. 生成挑战。

预期结果：

- 生成成功。
- 新 challenge 数据中包含 `xorTags` 字段。
- 备选答案不会导致创建失败。

### 3. 超长链接降级到 JSON

1. 填写更多问题和较长文本，尽量让 challenge 变大。
2. 生成挑战。

预期结果：

- 当链接过长时，界面提示建议使用 JSON。
- 仍可下载 JSON。
- 不应生成明显不可用的超长分享链接按钮状态。

## 二、求解挑战（新 challenge）

### 4. 正确答案刚好达到门限

1. 使用新生成 challenge 进入求解页。
2. 只填写恰好 `threshold` 个正确答案，其余留空。
3. 点击“解锁秘密”。

预期结果：

- 成功解锁秘密。
- 结果应与原始秘密完全一致。
- 不应出现超时或“达到求解上限”的提示。

### 5. 正确答案超过门限

1. 填写超过门限数量的正确答案。
2. 点击“解锁秘密”。

预期结果：

- 成功解锁秘密。
- 不因为填写更多正确答案而失败。

### 6. 少于门限

1. 只填写少于门限数量的答案。
2. 点击“解锁秘密”。

预期结果：

- 明确提示“至少需要回答多少题”。
- 不出现崩溃或异常错误。

### 7. 部分正确、部分错误

1. 填写达到门限数量，但其中至少 1 个答案是错误的。
2. 点击“解锁秘密”。

预期结果：

- 解锁失败。
- 提示应为“部分答案可能不正确”之类的验证失败信息。
- 对新 challenge，不应轻易触发超时或组合上限提示。

### 8. 多备选答案命中任一变体

1. 对带备选答案的问题，分别输入不同的正确变体。
2. 每种变体都尝试一次解锁。

预期结果：

- 任一合法备选答案都能通过该题验证。
- 不同变体不应导致不同的错误结果。

## 三、Solve 页状态机

### 11. 从 URL hash 自动加载

1. 打开包含 challenge hash 的链接。
2. 观察是否自动进入 Solve 页并显示题目。

预期结果：

- 自动加载成功。
- 不需要手动再点“加载”。

### 12. 清空 hash 后回到导入态

1. 已通过 hash 加载 challenge。
2. 手动删除地址栏 `#` 后的 challenge 数据并回车。

预期结果：

- 页面回到导入态。
- 旧 challenge 内容被清空。
- 不应继续显示旧题目。

### 13. 非法 hash 提示

1. 修改 URL hash 为无效内容。
2. 刷新或回车。

预期结果：

- 页面显示“加载失败”提示。
- 不应残留上一份 challenge 的内容。

### 14. 粘贴链接导入

1. 在 Solve 页点击“粘贴链接”。
2. 粘贴一个有效链接并点击“加载”。

预期结果：

- challenge 成功载入。
- 地址栏同步为对应 hash。
- 可刷新后继续保留该 challenge。

### 15. 文件导入

1. 在 Solve 页导入 JSON challenge。
2. 导入成功后刷新页面。

预期结果：

- challenge 可正常载入。
- 若该 challenge 来源于文件导入，刷新后不应错误地依赖旧 hash 自动恢复。

### 16. 更换挑战按钮

1. 已加载任意 challenge。
2. 点击“更换挑战”。

预期结果：

- 返回导入态。
- 题目、结果、错误提示、已填答案被清空。
- 如果当前 challenge 来自 hash / 粘贴链接，地址栏 hash 应一并清空。

## 四、国际化与文案

### 17. 文案定位

1. 观察首页标题、副标题、页脚、Solve 导入提示。
2. 切换语言为 `en` / `ja` / `zh-CN` / `zh-TW`。

预期结果：

- 文案统一表达为“离线问答解锁器”语义。
- 导入区提示明确说明“离线运行”和“弱答案可能被离线猜解”。

### 18. 语言切换后的 Solve 页

1. 先加载 challenge。
2. 再切换语言。

预期结果：

- 题目输入框和按钮文案会同步更新。
- 已填写答案不会丢失。

## 五、异常输入

### 19. 非法 JSON 文件

1. 导入一个格式错误的 `.json` 文件。

预期结果：

- 提示“无效的挑战文件格式”或等价错误。
- 页面状态保持稳定。

### 20. 空链接/无 hash 链接

1. 在“粘贴链接”区域输入空字符串，或输入不包含 `#` 的普通 URL。

预期结果：

- 给出明确错误提示。
- 不应加载出伪 challenge。

## 本轮最关键观察点

本轮改动后，最关键的是确认：

1. **新 challenge** 使用正确答案时，不再轻易触发组合爆炸带来的失败。
2. Solve 页在 hash 删除、hash 损坏、切换 challenge 时状态一致。
3. 新 challenge 的求解路径不再依赖组合搜索。
