# ShardKey

[English](README.md)

> 把秘密分享出去，并且只在“足够多的人”或“足够多的正确答案”同时到位时恢复它。

ShardKey 是一个基于浏览器的门限秘密分享工具。

它完全在客户端运行，支持离线使用，不需要构建流程、后端服务或外部依赖；直接打开 `index.html` 就能使用。

## 为什么是 ShardKey

很多秘密分享工具更偏向工程体系、基础设施或正式的密钥托管场景。

ShardKey 则刻意做得更轻：

- 打开浏览器就能用
- 可以直接分享链接或 JSON 文件
- 方便向非技术用户解释和交付
- 可以完全离线运行，包括 `file://`

它更适合“实用的恢复流程”，而不是“完整的秘密管理平台”。

## 两种恢复模式

### `经典问答`

把一个秘密放在“门限问答”后面。

例如：

- 一共 5 道题
- 只要答对其中 3 道就能解锁

这个模式适合“知道足够多背景信息、共同记忆或私有事实的人”来恢复秘密。

### `多人合力恢复`

把一个秘密拆成 `N` 份分片，至少收集 `K` 份后恢复。

如果说 `经典问答` 更像“闯关答题”，那 `多人合力恢复` 就有点像“集齐七龙珠召唤神龙”：单个分片本身只是碎片，但一旦收集到足够多、而且彼此有效的分片，秘密就能在浏览器本地重新出现。

这个模式适合把信任分散到多人、多设备或多个存储位置。

## 核心特点

- 所有处理都在浏览器本地完成
- 零外部依赖，不需要服务端
- 同时支持 `file://` 直接打开和静态托管
- 支持自包含链接与 JSON 导入导出
- 单个问题支持多个可接受答案
- 浏览器支持时可使用 Web Worker 后台处理
- 内置多语言界面：English、日本語、简体中文、繁體中文

- 用一个静态页面打包完整的秘密恢复体验

## 它底层是怎么工作的

从原理上说，ShardKey 是把几种成熟的密码学积木组合成一个更容易落地的浏览器恢复流程。

### 1. 先在本地加密秘密

你的秘密会先在浏览器里使用 Web Crypto API 通过 `AES-256-GCM` 加密。

### 2. 再把随机内容密钥做门限拆分

ShardKey 会生成一个随机密钥，并使用 `Shamir Secret Sharing` 在 `GF(256)` 上把它拆成多份 share。

这意味着：

- 少于门限数量的 share，不足以恢复密钥
- 任意一组达到门限的有效 share，都可以重建密钥

### 3. 不同模式，用不同方式交付 share

在 `经典问答` 模式下：

- 每个答案会先做规范化处理
- 规范化后的答案会经过 `PBKDF2-SHA-256` 强化
- 当前默认成本是每题 `120,000` 次迭代
- 每个正确答案可以解开一个被加密的 share

在 `多人合力恢复` 模式下：

- 每个分片直接携带一个门限 share
- 收集到足够多的有效分片后即可恢复

### 4. 全程离线恢复

当系统拿到足够多的有效 share 后，会在本地重建原始密钥，再用它解密出秘密。

由于秘密是通过 `AES-256-GCM` 封装的，恢复不仅要求“能解密”，还要求认证解密校验通过：如果重建出的密钥不对，或者恢复数据被破坏 / 篡改，完整性校验就会失败，秘密不会被接受。

生成和恢复都不依赖服务器。

## 安全理念

ShardKey 的定位是“实用的离线恢复工具”，而不是经过审计的专业秘密管理系统替代品。

它的优势：

- 所有处理都留在本地浏览器
- 分享数据中不包含明文秘密
- 使用现代浏览器原生密码学能力
- 不依赖后端也能实现门限恢复

需要注意的地方：

- 弱答案仍然可能被离线猜解
- 问答模式的安全性本质上取决于答案强度
- 高价值或长期保存的秘密，更适合使用密码管理器或经过审计的秘密共享工具
- 如果分享链接过长，不适合实际传递时，优先使用 JSON 导出

## 实用安全建议

- 对长期保存或影响较大的场景，优先使用高熵秘密
- 如果需要，可以在秘密后追加一段随机后缀来提高熵
- 如果希望恢复必须获得更多人同意，优先选择更高的门限值
- 单题的备选答案应尽量少、尽量不显眼，因为每增加一个可接受变体，都会略微扩大猜测面
- 当链接过长、不利于可靠传递时，优先使用 JSON 导出

## ShardKey 不是什么

ShardKey 不是：

- 托管式 secrets manager
- 企业级、经过审计的密钥托管系统替代品
- 对糟糕答案选择的兜底保障
- 适合永久保存最高价值秘密的最终方案

如果你需要强运营控制、访问日志、密钥轮换流程，或者硬件级密钥保护，请使用专门的系统。

## 实用限制

当前内置限制包括：

- 秘密长度：最多 `1024` 字节
- 问题数量：最多 `64`
- 门限值：最多 `64`
- 分片总数：最多 `20`
- 单题答案变体：最多 `16`

## FAQ

### 答案区分大小写吗？

不区分。答案在参与派生前会先做规范化处理，包括大小写折叠和空白整理，以减少纯格式差异导致的恢复失败。

### 一个问题可以设置多个正确答案吗？

可以。每个问题都可以配置多个可接受答案，只要命中其中任意一个就算答对。

### 分享出去的数据里包含明文秘密吗？

不包含。链接或 JSON 中承载的是加密后的恢复数据，而不是明文秘密本身；其中也不包含正确答案或答案哈希。

### 什么时候应该用 JSON，而不是链接？

当分享数据过大、URL 不方便传递，或者你的流程更适合文件分发时，就优先使用 JSON。

### 可以完全不依赖服务器运行吗？

可以。ShardKey 支持直接通过 `file://` 打开，也支持放到任意静态托管环境中运行。

## 开发说明

ShardKey 是一个轻量静态 Web 应用，日常开发不需要构建流程。

项目文件：

- `index.html` — 应用入口
- `css/style.css` — 界面样式
- `js/core.js` — 加密、分片与恢复核心逻辑
- `js/app.js` — 浏览器端交互逻辑
- `js/i18n.js` — 多语言与文案
- `js/worker.js` — 后台 Worker
- `scripts/smoke.mjs` — 冒烟测试

验证命令：

```bash
node --check js/core.js
node --check js/app.js
node --check js/i18n.js
node --check js/worker.js
node scripts/smoke.mjs
```
